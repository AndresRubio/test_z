# Two-model split: tiny Judge, larger generator

The topicality guardrail (Judge) runs as a prompt-only check on a small model
(gemma4:e2b) before retrieval; answers are generated by a larger model
(gemma4:e4b).

Why not the cheaper option — one model doing both in a single prompt:

- Guardrail leaks become invisible inside a generation prompt.
- Every off-topic query pays full generation latency.
- The split makes the guardrail independently testable and demonstrates
  right-sizing models per pipeline stage.

## Consequences

- Both models resident in Ollama together (~17 GB); fine on 32 GB RAM, worth
  noting for smaller machines.
- Two `ollama pull` lines in setup instead of one.
- Judge verdicts are structured output, so the guardrail can be evaluated
  standalone with a labeled query set.
